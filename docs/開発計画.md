# 開発計画（並列開発向け）

Claude Code を並列で動かして最速で作るための実装手順書。
**どのタスクを最初に直列でやり、どこから並列に分岐できるか**を依存関係から設計している。

---

## 全体像

```
STEP 1 基盤（直列・全体をブロック）
┌──────────────┬──────────────┬──────────────┐
│ 1-A アプリ基盤 │ 1-B Supabase  │ 1-C 共有モデル │  ← この3つは並列OK
└──────────────┴──────────────┴──────────────┘
        ↓ 全部 dev にマージ（= 全員の作業起点）
        │
        │  ※ 技術検証スパイク（S）は STEP1 と並行で早期に回す
        ↓
STEP 2 機能開発（4レーン並列）
┌──────────┬──────────┬──────────┬──────────┐
│ A 認証     │ B ホーム   │ C グループ  │ D 投稿     │
│ プロフィール │           │ 作成/参加/詳細│ 撮影/送信   │
└──────────┴──────────┴──────────┴──────────┘
        ↓ 各レーンを dev にマージ
        ↓
STEP 3 結合・E2E動作確認（直列）
        ↓
STEP 4 配布（APKビルド・実機確認）
```

**クリティカルパス**: `STEP1 → STEP2の一番重いレーン → STEP3 → STEP4`
最も重いのは Lane D（投稿：カメラ）と Lane C（グループ：画面3つ）。
ここに人を厚めに配置すると全体が縮む。

---

## なぜこの順序なのか

### STEP1 を必ず最初に直列でやる理由

全員が同じ「骨組み」から branch を切る必要があるため。
ルーター・DI・フォルダ構成・モデルが無い状態で各自が作り始めると、
後で必ず構造が衝突する。**ここを1回固めれば、以降は衝突しない。**

### STEP2 が並列にできる理由

feature-first 構成なので、各レーンは自分の `features/〇〇/` の中だけを触る。
共有する `models/` と `core/router.dart` は STEP1 で確定済みなので、
基本的に読むだけ。→ **ファイル単位で競合しない**ので安全に並列化できる。

### レーン間の「ゆるい依存」はシードデータで解消する

- ホーム / グループ詳細 → 投稿データを表示する
- 投稿（送信） → グループ一覧が必要

これらは互いの**画面の完成を待つ必要はない**。
Supabase に手動 or SQL でテスト用データ（シード）を入れておけば、
各レーンは独立して開発・動作確認できる。

---

## STEP 1：基盤（最優先・直列）

> このSTEPが dev にマージされるまで、STEP2 は開始しない。
> 1-A / 1-B / 1-C は別々の人 or インスタンスで**並列可能**。

### 1-A アプリ基盤
- `flutter create` でプロジェクト初期化
- `pubspec.yaml` に依存追加
  （flutter_riverpod / go_router / supabase_flutter / flutter_dotenv / camera / video_player / google_sign_in）
- フォルダ構成のスケルトン作成（[コーディング規約](./コーディング規約.md)準拠）
- `main.dart`：ProviderScope + dotenv.load + Supabase.initialize + router
- `core/supabase_client.dart`
- `core/router.dart`：**全画面のルートをプレースホルダー画面で先に定義**
  （各画面は「〇〇画面」とだけ表示する仮Widget）

**完了の定義**：ビルドが通り、プレースホルダーで全画面を遷移できる。

### 1-B Supabaseバックエンド
- Supabase プロジェクト作成
- テーブル作成（[データベース設計](./データベース設計.md)準拠）
- 制約（invite_code UNIQUE、group_members UNIQUE、post_shares の1時間制限用）
- RLS ポリシー設定
- Storage バケット作成（videos / icons）
- `.env` に URL / anon key を設定し、チームへ共有
- SQL を `supabase/schema.sql` としてリポジトリに保存（再現用）

**完了の定義**：DBにテスト用データを入れて、各テーブルが期待通り読める。

### 1-C 共有モデル
- `models/user.dart` / `group.dart` / `post.dart` / `post_share.dart`
- 各モデルに `fromJson` / `toJson`
- データ取得は各 feature の Provider が Supabase を直接呼ぶ方針
  （共有リポジトリ層は作らない＝レーン間のファイル競合を避けるため。
  　数クエリの重複は許容する）

**完了の定義**：モデルが Supabase のレスポンスをパースできる。

---

## 技術検証スパイク（S）※ STEP1 と並行・早期に

カメラの Flutter Web 対応は[確認事項](./開発・配布・運用.md)で未検証。
**ダメだった場合に方針転換が必要なので、最優先で潰す。**

- Flutter Web でカメラ起動できるか
- Flutter Web で動画撮影できるか
- Flutter Web で Supabase Storage にアップロードできるか
- ダメな場合の代替案（Android実機のみで開発 / `file_picker` で動画選択 等）

**結果を [進捗管理.md](./進捗管理.md) に記録する。** Lane D の進め方がこれで決まる。

---

## STEP 2：機能開発（4レーン並列）

各レーンは dev から `feat/〇〇` ブランチを切って独立に進める。

### Lane A：認証・プロフィール（`features/auth/`）
- スプラッシュのログイン判定 → 分岐
- ログイン画面（メール / Google）
- プロフィール入力画面（初回・名前登録）
- `auth_provider`（セッション監視）
- users テーブルへの登録
- **最も独立性が高い**（他レーンに依存しない）

### Lane B：ホーム（`features/home/`）
- ホーム画面レイアウト
- 自分のVlog一覧（posts 時系列）
- 参加中グループ一覧
- 各ボタンの遷移（ログ作成 / グループ作成 / 参加 / 詳細）
- `home_provider`
- **画面遷移のハブ**。遷移先はSTEP1のプレースホルダーで先に繋がる。

### Lane C：グループ（`features/group/`）
- グループ作成（招待コード生成・自動参加）
- グループ参加（招待コード）
- グループ詳細レイアウト
- 時間移動（左右タップ・1時間）
- 日付移動（左右スワイプ・1日）
- メンバー一覧 / 招待コード表示 / グループ退出
- `group_provider`
- **画面数が多い**ので重いレーン。

### Lane D：投稿（`features/post/`）
- 撮影画面（自動起動・撮影・イン/アウト切替・タイマー）
- 送信画面（プレビュー・取消・送信）
- 送信先グループ選択（**1時間制限＝グレーアウト**、[投稿ルール](./投稿ルール.md)）
- Storage アップロード → posts + post_shares 書き込み
- `post_provider`
- **技術リスクが高い**（カメラ）。スパイク(S)の結果を受けて進める。

---

## STEP 3：結合・E2E（直列）

- プレースホルダーを外して本物のナビゲーションに結線
- E2Eシナリオ確認：
  **ログイン → グループ作成 → 撮影 → 送信 → グループ詳細で表示**
- 1時間制限のグレーアウトが効くか確認
- Vlog一覧が時系列で出るか確認
- バグ修正

---

## STEP 4：配布

- GitHub Actions ワークフロー作成（`.github/workflows/build.yml`）
- APK ビルド成功
- Android 実機にインストールして動作確認
- [発表前チェックリスト](./開発・配布・運用.md)を消化

## 並列インスタンスの割り当て例（4人想定）

| 人 | STEP1 | スパイク | STEP2 |
|----|-------|---------|-------|
| 1（リード） | 1-A アプリ基盤 | — | Lane B ホーム |
| 2 | 1-B Supabase | — | Lane C グループ |
| 3 | 1-C モデル | S 技術検証 | Lane D 投稿 |
| 4 | （1-A補助 / レビュー） | — | Lane A 認証 |

- STEP1 完了 → dev マージ → 全員ここから branch
- 人数が少ない場合は、1人が複数レーンを順番に持つ（クリティカルパスの C・D を優先）

---

## 並列開発のルール

- 作業は必ず `feat/〇〇` ブランチで（dev へ直プッシュ禁止）
- 自分のレーンの `features/〇〇/` 以外は基本触らない
- `core/router.dart` / `models/` を変更したいときは**チームに一声かける**（共有ファイルなので）
- 着手・完了したら [進捗管理.md](./進捗管理.md) を更新する
- ブロッカーが出たら進捗管理ファイルの「ブロッカー」に書く
