## 技術検証スパイク（S）　担当：Claude（Lane D）

> 結果次第で Lane D の進め方が変わる。最優先で潰す。

- ✅ Flutter Web でカメラ起動
- ✅ Flutter Web で動画撮影
- ✅ Flutter Web で Storage アップロード
- ✅ ダメな場合の代替案を決める（→ 不要。camera パッケージがWebで動くため代替案は採用せず）

**検証結果メモ：**

- `camera: ^0.12.0+1` は Web 公式実装 `camera_web 0.3.5+4` を内包しており、Flutter Web で動作する。
  - `flutter build web` 成功、`flutter run -d web-server --web-port=8769` で起動・配信を確認。
  - `availableCameras()` / `CameraController.initialize()` / `startVideoRecording()` / `stopVideoRecording()` がWebでも利用可能（内部は MediaRecorder）。
- **camera をそのまま採用**。`file_picker` 等の代替案は不要。
- 実運用上の注意（Webのみ）:
  - カメラ取得は**セキュアコンテキスト必須**（`localhost` か HTTPS）。`web-server` の localhost はOK。LAN内の他端末からIPアクセスする場合はHTTPSが必要。
  - 初回アクセス時にブラウザのカメラ／マイク許可ダイアログが出る（要ユーザー操作）。
  - Web の録画形式は **webm**（モバイルは mp4）。Storage の拡張子・contentType は録画ファイルから判定して保存している。
  - 動画プレビューはプラットフォーム差を `video_preview_factory.dart`（条件付きインポート）で吸収（Web=blob URL / モバイル=File）。

## メモ・決定事項

> 開発中に決めたこと、共有事項をここに残す。

- モデルの User は `AppUser` という名前にした（Supabase Auth の `User` と衝突するため）。usersテーブル対応。
- データ取得は各 feature の Provider が Supabase を直接呼ぶ方針（共有リポジトリ層は作らない）。
- ルートは `core/router.dart` に集約済み。各画面はプレースホルダーが入っているので、担当はそのファイルを差し替えるだけでよい。
  - 画面ファイルの場所：`features/auth/`（splash/login/profile）, `features/home/`, `features/group/`（create/join/detail）, `features/post/`（camera/send）
- ✅ STEP1 完了。`flutter run -d web-server` でスプラッシュ表示・画面遷移を確認済み。
- **【Web版の必須対応】** `supabase_flutter` はWeb初期化時に Passkeys Web SDK（`bundle.js`）を要求する。
  無いと `Supabase.initialize` が例外を投げて画面が真っ白になる。
  対応：`web/passkeys_bundle.js`（corbado flutter-passkeys 2.4.0）を配置し、`web/index.html` の `<head>` で読み込む。
  ※ APKビルド（Android）では不要だが、Web確認では必須。
- 起動時の初期化エラーは `main.dart` の try/catch で画面に赤字表示されるようにしてある（真っ白防止）。
