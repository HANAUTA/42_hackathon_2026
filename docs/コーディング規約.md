# コーディング規約

## フォルダ構成

```
lib/
├── main.dart
├── core/
│   ├── supabase_client.dart   # Supabase初期化
│   └── router.dart            # 画面遷移定義
├── features/
│   ├── auth/
│   │   ├── login_screen.dart
│   │   └── auth_provider.dart
│   ├── home/
│   │   ├── home_screen.dart
│   │   └── home_provider.dart
│   ├── group/
│   │   ├── group_detail_screen.dart
│   │   ├── group_create_screen.dart
│   │   └── group_provider.dart
│   └── post/
│       ├── camera_screen.dart
│       ├── send_screen.dart
│       └── post_provider.dart
└── models/
    ├── user.dart
    ├── group.dart
    └── post.dart
```

- 機能単位で `features/` 以下にフォルダを切る
- 複数機能で共通して使うモデルは `models/` に置く
- Supabase初期化・ルーティングなどアプリ全体の基盤は `core/` に置く

---

## 状態管理

- **Riverpod**（`flutter_riverpod`）を使う
- 状態は Provider に集約し、画面（Widget）に直接ロジックを書かない
- Provider は `features/〇〇/〇〇_provider.dart` に置く

---

## ルーティング

- **go_router** を使う
- 全ルートは `core/router.dart` に集約する
- 画面内で直接 `Navigator.push` は使わない

---

## Git運用

- ブランチ: `feat/機能名`（例: `feat/グループ詳細`）
- mainへの直プッシュ禁止、dev経由でPR
- コミットメッセージ: 日本語OK（例: `グループ詳細画面の時間移動を実装`）

---

## 環境変数

- Supabase の URL / anon key は `.env` に書く
- `.env` は Gitに上げない（`.gitignore` 済み）
- チームに共有するときは `.env.example` をコピーして値を入れる
- 読み込みには `flutter_dotenv` を使う

```dart
// core/supabase_client.dart
await dotenv.load(fileName: '.env');
await Supabase.initialize(
  url: dotenv.env['SUPABASE_URL']!,
  anonKey: dotenv.env['SUPABASE_ANON_KEY']!,
);
```

## 命名規則

| 対象 | 規則 | 例 |
|------|------|----|
| ファイル名 | snake_case | `group_detail_screen.dart` |
| クラス名 | PascalCase | `GroupDetailScreen` |
| 変数・関数名 | camelCase | `fetchGroupPosts()` |
| Provider | camelCase + Provider | `groupProvider` |

---

## コメント規則

### 書く場所

| 場所 | ルール |
|------|--------|
| ファイル先頭 | **必須**・1〜2行・そのファイルの責務を日本語で書く |
| クラス・Provider | 任意・1行・役割を日本語で書く |
| メソッド内 | 基本なし。「なぜそう書いたか」が分かりにくい場合のみ |
| 処理の行単位 | 書かない |

### 書かない内容

- 「このコードは〇〇をしています」→ コードを読めば分かる
- 関数名・変数名で分かること

### 書く内容

- 「このファイルは〇〇を担当しています」
- 「このProviderは〇〇の状態を管理します」
- なぜその実装にしたか（制約・理由）
